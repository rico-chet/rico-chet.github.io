Mocking a script with pure bats

Abstract

Here, I will show how a script can be tested in regard to its collaborations with other executables.

Production

Figure 1. Sequence in production

Goal

Ensure that robot.sh calls legs.sh as expected.

Constraint

Don’t modify existing code or its structure.

Solution

Overview

Replace calls to legs.sh with calls to another script which logs its parameters. Then, assert that the calls are the expected ones.

Figure 2. Sequence in test (overview)

Detailed

bats = Bash Automated Testing System is a convenient test framework for shell scripts and other executables.

The original bats was abandoned in 2016, but its development continued as bats-core by a new team.

Figure 3. Sequence in test (detailed)

Challenges

  • There can be multiple tests in a *.bats file and running these in parallel requires careful coding.

  • In Debian 9 and 10, only the old (0.4) version of bats is available out of the box, and supporting both current (1.*) and legacy bats is additional effort. Thus, a decision to only use the current bats (also known as bats-core) can save future developer effort at the expense of additional initial effort.

  • Boilerplate code is plenty, but could be simplified when using one of the two bats-mock libraries.
